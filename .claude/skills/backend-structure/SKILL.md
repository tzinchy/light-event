# Backend: структура и миграции

## Нарезка по сущностям (vertical slice), не по слоям

Каждая доменная сущность — своя папка `app/<entity>/`, внутри — файлы по ответственности:

```
apps/api/app/
  <entity>/            # напр. user, company, filial, vacancy, application, test, balance, document, chat, review, complaint
    router.py          # FastAPI-эндпоинты сущности
    service.py         # бизнес-логика (service)
    repo.py            # доступ к БД (repository), только этот слой знает про SQLAlchemy
    schemas.py         # Pydantic-схемы (in/out)
    models.py          # SQLAlchemy-модель(и) сущности
  core/                # конфиг, зависимости, security, session, общие утилиты
  main.py              # сборка приложения, подключение роутеров
apps/api/tests/
  <entity>/            # тесты зеркалят структуру сущностей
```

Правила:

- service содержит логику и оркестрацию; repo — только запросы к БД; schemas — валидация/сериализация. Роутер тонкий: разбирает запрос → вызывает service → отдаёт schema.
- Сервис не лезет в БД напрямую в обход repo. Роутер не содержит бизнес-логики.
- Межсущностные операции (напр. публикация вакансии списывает баланс) — в service, который вызывает repo нужных сущностей в одной транзакции.

## Alembic — только командами, структуру руками НЕ городить

- Не создавать вручную дерево миграций и не писать ревизии с нуля.
- Инициализация — стандартной командой `alembic init`, дальше только:
  - `alembic revision --autogenerate -m "<описание>"` — сгенерировать миграцию из моделей;
  - `alembic upgrade head` — применить;
  - `alembic downgrade -1` — откатить при необходимости.
- Никакого `Base.metadata.create_all()` в приложении и тестах — схема поднимается только через `alembic upgrade` (в т.ч. в testcontainers).
- После изменения моделей — сгенерировать ревизию автогеном, просмотреть diff, при необходимости поправить, применить.
