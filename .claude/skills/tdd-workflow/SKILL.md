# TDD: red → green → refactor

1. **Сначала падающий тест.** Ни строчки бизнес-логики без покрывающего теста. Тест пишется ПЕРВЫМ и должен упасть по содержательной причине (не из-за опечатки/импорта).
2. **Минимальный код до зелёного.** Реализация ровно под тест, без спекулятивных обобщений.
3. **Рефактор на зелёном.** Улучшать структуру можно только когда все тесты зелёные.
4. **Не переходить к следующему модулю**, пока текущий не зелёный целиком.

Правила:

- Тесты backend — pytest + testcontainers (реальные Postgres/Redis/MinIO, см. skill testcontainers). Никаких SQLite-подмен и моков БД.
- Мокается только внешний мир за интерфейсами (SmsProvider, OAuthProvider, PaymentProvider) — не данные и не своя БД.
- На каждую доменную операцию — позитивный и негативные сценарии (запрет доступа, недостаток средств, неверный статус-переход).
- Frontend: Vitest + Testing Library для логики/компонентов, Playwright для e2e.
- Фикстуры данных живут только внутри тестов и не попадают в dev/prod БД.
